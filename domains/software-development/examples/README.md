# Examples

This directory contains real examples of CONSTITUTION.md files and sample outputs from the AI-Dev-Orchestrator framework.

## Complete CONSTITUTION.md Examples

### [Internal Tool: Team Expense Tracker](./internal-tool/CONSTITUTION.md)
A realistic example for a 10-person sales team expense tracking tool. Shows:
- Simplified requirements for internal use
- Practical tech stack choices (Next.js, Supabase, Tailwind)
- Focus on function over form
- Clear scope limitations

**Best for:** Understanding how to configure for small internal tools

---

### [Client App: SaaS Analytics Dashboard](./client-app/CONSTITUTION.md)
A production-grade example for a customer-facing SaaS product. Shows:
- Comprehensive security requirements
- Testing and monitoring strategies
- Performance targets and optimization
- Professional quality standards

**Best for:** Understanding production-grade configurations

---

### [AI Agent: Slack Customer Support Bot](./ai-agent/CONSTITUTION.md)
An API-focused example for an always-on service. Shows:
- Bot-specific patterns (retries, rate limiting, error handling)
- Integration management (OpenAI, Slack, Pinecone)
- Monitoring and observability
- Reliability-first approach

**Best for:** Understanding bot and agent-specific requirements

## How to Use These Examples

### If you're building an internal tool:
1. Read the [internal tool example](./internal-tool/CONSTITUTION.md)
2. Copy `CONSTITUTION-TEMPLATE.md` from `/templates/internal-tool-constitution.md`
3. Customize it based on the example

### If you're building a client-facing app:
1. Read the [client app example](./client-app/CONSTITUTION.md)
2. Copy `CONSTITUTION-TEMPLATE.md` from `/templates/client-app-constitution.md`
3. Customize it based on the example

### If you're building a bot or agent:
1. Read the [AI agent example](./ai-agent/CONSTITUTION.md)
2. Copy `CONSTITUTION-TEMPLATE.md` from `/templates/ai-agent-constitution.md`
3. Customize it based on the example

### To understand the workflow outputs:
1. Read the [sample PRD](./sample-outputs/sample-prd.md) to see what Phase 1 produces
2. Read the [sample tech spec](./sample-outputs/sample-tech-spec.md) to see what Phase 2 produces
3. Use these as reference when working through your own feature
